4.3 Cluster Hygiene — The numpy<2 Pin

Two install-time landmines, both real, both documented in requirements.txt
  • Do NOT pin lxml. DBR 14.x ships a native-extension lxml. Forcing an upgrade replaces the binary with an ABI-mismatched wheel and crashes the kernel on restartPython() with exit code 1 and empty stdout. sec-parser handles lxml transitively — let it.
  • Do pin numpy<2. sec-parser declares pandas as a dependency, and pip's resolver will transitively upgrade NumPy to 2.x. DBR 14.x's pyarrow C extension is compiled against the NumPy 1.x ABI and fails to import under NumPy 2.x with AttributeError: _ARRAY_API not found, which crashes the kernel on boot when dbruntime.PipMagicOverrides imports pandas → pyarrow. Remove this pin only when DBR ships a NumPy-2.x-compatible pyarrow (DBR 16+).

Notebook 03 enforces both with a single MAGIC line: %pip install --quiet sec-parser "numpy<2" followed by dbutils.library.restartPython(). The cell stays pure-MAGIC — mixing %pip with Python in the same cell silently breaks library install on serverless.

4.4 Section Extraction — Why sec-parser Replaced the Regex Pipeline

The original regex extractor on entity-decoded flat text was 100% broken on AMZN and V (~50 filings each yielded zero sections) and produced thousands of partial failures across the other 28 tickers. Three independent root causes:

  • HTML-entity blindness — raw filings contain &#160;, &#8217;, &mdash; etc. inside Item anchors. The flat-text path matched on raw bytes, so item\s+1\b never fired against Item&#160;1.
  • iXBRL span fragmentation — "Item" and "1A" routinely sit in separate inline-XBRL spans. Flat-text concatenation joins them with no whitespace boundary, breaking the regex.
  • Page-footer leakage — static page numbers and copyright footers landed inside section bodies, inflating word counts and polluting embeddings.

sec-parser operates on the iXBRL DOM tree, so all three failure modes disappear at the source. The notebook keeps the regex extractor as a per-section fallback for the small set of filers (notably JPM, BAC, MA, V 10-K MD&A and Risk Factors, MSFT 10-Q MD&A) where sec-parser's classifier promotes some Items to TopSectionTitle but mis-classifies others. The merge is per-section: sec-parser wins ties; regex fills only the still-missing required sections. Each row carries extractor_used ∈ {sec-parser, regex-fallback} so downstream filtering can scope by extractor.

Two further fixes round out the extractor:

  • SGML <TYPE> unwrap. A full-submission.txt contains many <DOCUMENT> blocks (cover page, exhibits, XBRL schemas). The unwrap matches by <TYPE> to pick the 10-K or 10-Q body deterministically. The previous "first document wins" heuristic could grab a cover-page iXBRL doc and silently truncate the filing.
  • Heading-preference rule. When sec-parser exposes both a TOC stub (a bare "Item 2" <= 12 chars) and a descriptive full heading, the extractor prefers the substantive one. This single rule recovered all 19 MSFT 10-Q MD&A extractions; first-match-wins had been picking the stub and producing near-empty bodies that failed the 150-word minimum.

4.5 Financial Extraction Hierarchy — The LLM Never Extracts Numbers

Hard rule, encoded in the SYSTEM_PROMPT

Numbers come from XBRL. Text comes from filings. The agent must use a tool for every figure it cites — it is not allowed to read a number out of a retrieved chunk and present it as fact.

LayerWhatUsed for
L1 XBRL (system of record) Every numerical metric in company_metrics and company_metrics_quarterly. Sourced from CompanyFacts JSON via TARGET_CONCEPT_MAP (29 raw concepts → 12 normalised line items).
L2 sec-parser DOM extraction Filing text only: Business, Risk Factors, MD&A, Risk Factors Updates. Drives the vector index.
L3 Concept-priority + canonical-accession selection Deterministic dedup: when a filer tags both RevenueFromContractWithCustomerExcludingAssessedTax and SalesRevenueNet, priority 1 wins. Canonical accession per (ticker, fy) chosen by metric coverage + latest filing date.
L4 LLM (reasoning only) Synthesises tool outputs into prose. Required to label [VERBATIM] vs [SUMMARY] on every quote and disclose any ratio formula on first use.

Cross-validation is built into the Gold layer itself: data_quality_score = (count of populated base metrics) / 9, materialised on every row. Tickers and years that fall below the threshold are visible in a single SQL query and are flagged in the agent's output.

Q2/Q3 derived by YTD subtraction. Most filers tag only the cumulative concepts (SixMonthsEnded, NineMonthsEnded) in Q2/Q3 10-Qs, not the discrete 90-day variant. Each fact is classified as discrete (80–100d), ytd_6mo (170–195d), ytd_9mo (260–285d), or instant. Then:
  Q1 = discrete
  Q2 = coalesce(Q2_90d_tagged, Q2_YTD_6mo − Q1_90d)
  Q3 = coalesce(Q3_90d_tagged, Q3_YTD_9mo − Q2_YTD_6mo)

6.1 Chunking Strategy — What's Wired and Why

DecisionValueWhy
Tokeniser tiktoken for text-embedding-3-large, fall back to cl100k_base Token-accurate chunking that aligns with the embedding model's window. The fallback keeps the chunker working when the model name is not in tiktoken's encoding table.
Chunk size / overlap 512 tokens / 64-token overlap (chunk_version = tok_512_64_v1) 512 fits comfortably inside the BGE-large-en context (8K), gives ~3× smaller index than 1024 and ~2× better passage focus than 1024 for financial-text queries. 64 tokens of overlap (12.5%) preserves cross-paragraph continuity at item-boundary transitions without exploding chunk count.
Chunk ID sha256(canonical-json({filing_id, section_name, chunk_index, chunk_text, chunk_version})) Re-running the chunker produces identical IDs for unchanged sections. Crucially, the hash includes chunk_text: any upstream silver change (sec-parser swap, heading-preference rule) regenerates the IDs — which is what triggers the delete-then-append idempotency below.
Idempotency Per-(filing_id, section_name) MERGE-delete then APPEND A plain MERGE on chunk_id would leave the prior run's chunks side-by-side with the new ones (different IDs, both insert as new rows). Deleting every chunk for the affected (filing_id, section_name) pairs first — then appending — is the only correct semantics for a content-addressed primary key. Unrelated filings (e.g. ticker-filtered runs) are untouched.
Pre-write quality guards RuntimeError on duplicate chunk_id, null/empty chunk_text, or non-positive token count Catches malformed UDF output before it lands in the chunks table and corrupts the index.
Filterable metadata ticker, section_name, fiscal_year, filing_type Lets the agent's search_filings tool scope retrieval surgically (e.g. AAPL FY2024 Risk Factors, 10-K only) and prevents multi-year chunk mixing on "most recent" questions.

6.2 Vector Search Index Provisioning — SDK Quirks

The notebook orchestrates: endpoint creation → endpoint ONLINE → index creation → index READY → explicit sync. Three production-grade behaviours are baked in:

  • Pre-flight source-table validation. _assert_source_table_quality verifies the primary key has no NULLs and no duplicates, and that the embedding source column has no null/empty rows. CDF status is logged for visibility. The provisioner refuses to call the VS API on a malformed source.
  • Bypass idx.sync(). The Vector Search SDK has a long-standing bug where idx.sync() blocks forever even after the server has accepted the request. The notebook POSTs directly to /api/2.0/vector-search/indexes/{name}/sync with a 10s socket timeout, then polls state. Bounded and provably non-hanging.
  • Liberal "ready" detection. Index state semantics drift across DBR versions. The poll accepts any of ONLINE, READY, ONLINE_NO_PENDING_UPDATE, or any state starting with ONLINE, plus messages containing "succeeded". Treats FAILED / ERROR / UNHEALTHY as terminal.

6.3 Retrieval — search_filings Tool

The agent's vector tool (06_rag_agent.py → search_filings) is the only retrieval surface in the system. Behaviours:

  • Filter set. ticker, section_name (enum: Business / Risk Factors / MD&A / Risk Factors Updates), fiscal_year, filing_type (enum: 10-K / 10-Q). All optional. The system prompt instructs the LLM to pass fiscal_year for any "most recent / latest" question to prevent multi-year chunk mixing.
  • Schema introspection at load time. _detect_index_columns reads the live VS index's schema_json and remembers which metadata columns are actually materialised. filing_type was added when 10-Q support landed; a DELTA_SYNC index only picks up new columns after a full re-provision, so the tool gracefully degrades on the older 4-column shape rather than 400-ing.
  • Similarity threshold gate. Default 0.6. Rows below the threshold are filtered out. If everything is below threshold the tool returns "No passages met the similarity threshold. Try a broader query." instead of fabricating a result.
  • Citation format. Every passage is prefixed with [Source: TICKER | FY{int(fy)} | FilingType | Section]. The fiscal_year cast to int matters — it's stored as DOUBLE in Delta and would otherwise render as FY2024.0.

6.4 Why Pure Vector + Filters (Not Hybrid)

The blueprint originally proposed a chunking experiment matrix and a hybrid (vector + BM25 + re-rank) search comparison. The implemented system is pure vector search with metadata filters. Two reasons that hold up under scrutiny:

  1. The query language is structured at the boundary. The agent calls search_filings with explicit ticker / section_name / fiscal_year / filing_type filters. These filters are exact — they are the keyword half of a hybrid system, just expressed as structured arguments instead of inline lexical operators. Adding BM25 on top would re-derive what the filters already give us deterministically.
  2. Free-text inside a filtered slice is short and topical. Once you've scoped to "AAPL FY2024 Risk Factors", you're searching ~30 chunks for one section's content. Pure semantic search dominates lexical at that scale. The places where BM25 helps — long, mixed-topic corpora with rare proper nouns — are not the regime here.

If the evaluation suite (notebook 07) shows retrieval-grounded scorers below threshold, the reranking + hybrid experiment is the natural next iteration — deferred until measured.

7.3 Predict Loop

messages = [{system_prompt}] + user_messages

for iteration in range(MAX_ITERATIONS=5):
    response = llm.predict(
        messages=messages,
        tools=TOOL_SCHEMAS,         # OpenAI function-calling format
        tool_choice="auto",
        temperature=0.1,
        max_tokens=2048,
    )

    if not response.tool_calls or finish == "stop":
        return final_content    # final answer reached

    messages.append(assistant_with_tool_calls)

    for tc in tool_calls:
        result = TOOL_DISPATCH[tc.name](tc.args)
        messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})

# Iteration cap reached -- ask the LLM for a best-effort answer
messages.append({"role": "user", "content": "Based on the tool results above, provide your best answer now."})
return llm.predict(messages=messages, temperature=0.1, max_tokens=1024)

Two design choices worth naming:

  • Temperature 0.1 — non-zero so the model can phrase comparisons fluently, low enough that tool selection is deterministic across reruns.
  • Hard 5-iteration cap with a final "best answer now" fallback — bounds latency and cost, prevents tool-call infinite loops, and the explicit final prompt forces synthesis instead of dangling mid-loop.

7.4 SYSTEM_PROMPT — Routing & Output Rules

Verbatim rules baked into the prompt
  • Annual / fiscal-year questions ("revenue in FY2024", "2023 operating margin") → get_company_metrics.
  • Interim / within-year questions ("Q2 2024 revenue", "how did margins change Q1–Q3") → get_quarterly_metrics. Q4 standalone is NOT available — fall back to annual totals and flag the limitation.
  • Qualitative questions (risks, strategy, management commentary) → search_filings.
  • "Most recent" / "latest" questions → first call the appropriate metrics tool to identify the latest year (and quarter if interim) for that ticker, then pass that fiscal_year into search_filings so retrieved passages all come from a single filing period.
  • Citation labelling — [VERBATIM] on direct quotes, [SUMMARY] on paraphrases. Never present a summary as a quote.
  • Formula disclosure — state the formula explicitly on first use (e.g. "Operating Margin (GAAP) = Operating Income ÷ Revenue").

7.6 Why This Stays a Single Agent

The original blueprint reserved the option to split into multiple agents if tool-selection accuracy fell below 80% on complex queries. Eight weeks in, the actual data shape doesn't motivate that split:

  • The three tools are orthogonal in input space (free text vs. ticker+year vs. ticker+year+quarter). The router collapses to a small decision tree the LLM handles cleanly with the prompt rules above.
  • The 5-iteration cap puts a hard ceiling on latency. Splitting into multiple agents would add inter-agent message passing without changing the underlying tool count.

The split decision rebooks once notebook 07's evaluation surfaces measured tool-selection accuracy — not before.

8.1 Why a 10-Question Curated Set, Not 100

The original blueprint specified a 100-question set. The implemented set is 10. Three reasons that matter:

  • Manual verification dominates the cost. Each question requires reading the actual filing and confirming the answer. 10 carefully verified questions beat 100 hastily verified ones — a single wrong "ground truth" answer poisons the LLM-as-judge.
  • The set is structured to exercise every tool path. 2 questions per category × 5 categories × 8 unique tickers covers numerical lookup, risk-text retrieval, multi-row SQL, multi-company comparison, and citation validation. Adding more questions in the same shape would change confidence intervals, not surface new failure modes.
  • Iteration speed. A full eval run hits the live agent endpoint sequentially. 10 questions is a 2-minute feedback loop; 100 is 20 minutes. Scaling to 100 is the natural next step once the scorer suite stabilises.

Two-Tool Infrastructure Split

Terraform owns long-lived infrastructure that outlasts any single deploy: cluster policies, secret scopes, service-principal lookup. The Databricks Asset Bundle (databricks.yml) owns job topology — the seven notebook tasks of finsage_daily_run, the shared job cluster, schedule, and notification config. The two tools are complementary, not competing: Terraform is reapplied rarely, the bundle on every push to main.

Notes on Things Deferred

  • No src/processing/, src/retrieval/, src/agent/. The Spark transformations live inside the notebooks (which are the deployable artifacts on Databricks); pulling them into pure-Python modules adds an indirection layer with no testability win at this scale. Pure-Python helpers that benefit from unit tests live under src/finsage/ and src/ingestion/.
  • No transcripts pipeline. The original blueprint included earnings transcripts; deferred to v2 once the SEC-filing path's eval signal stabilises.
  • No Streamlit app. The agent is reachable via w.serving_endpoints.query(); an app is a demo wrapper, not a system requirement. Will land in W9/W10 if eval iteration time allows.
  • No risk-indicators Gold table. Year-over-year risk diff is reachable today via two search_filings calls; a precomputed indicator table is a v2 enhancement.
