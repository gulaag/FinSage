4.4 Financial Extraction Strategy — Decide This Now

Hard Rule: The LLM Never Extracts Numbers

If I rely on an LLM to pull "revenue = $391B" from a filing document, numerical precision will collapse. LLMs approximate. Financial data must be exact. The extraction stack is layered:

LayerWhatUsed ForWhy
Layer 1 XBRL (Primary) All 12 target financial metrics (revenue, net income, total assets, etc.) Machine-readable structured data filed with the SEC. Exact numbers. No parsing ambiguity. This is my ground truth source.
Layer 2 Document Parsing (Secondary) Tables and data NOT in XBRL: segment breakdowns, supplemental tables, footnotes Use only for qualitative context, supplemental tables, or future edge cases where XBRL does not cover the requirement.
Layer 3 Normalization Mapping (Deterministic) "Net sales" → "revenue", "Cost of sales" → "cost_of_revenue" A lookup table, not an LLM call. Regex + exact matching. Manually verified for 30 companies.
Layer 4 LLM (Reasoning Only) Risk factor summarization, management tone analysis, multi-source synthesis The LLM reasons over already-extracted data. It never pulls raw numbers from documents.

Cross-validation: The current production path treats XBRL-backed CompanyFacts as the numerical system of record. If I later add document-extracted table values, they must be compared against silver.financial_statements; any mismatch is logged and the XBRL-backed value wins.

This decision affects my entire architecture. Write it into docs/technical_decisions.md on day one.

Every answer must cite its source. This makes my system enterprise-grade.

  • Answer grounding verification: Check that claims in the response are supported by cited chunks
  • Confidence scoring: Lower confidence when fewer sources found, or when sources disagree

8 Evaluation Strategy — The Career Differentiator

This Is the Single Most Important Section

Every other portfolio project on GitHub has a README that says "built a RAG system." Almost none have a rigorous evaluation suite with documented improvement iterations. This is what separates me from 1000 other applicants.

North Star Metric

Numerical Precision ≥ 92% on the benchmark set

The percentage of financial number queries where the agent returns the correct value (within ±1% tolerance), with a valid source citation. If the numbers are wrong, nothing else matters.

Supporting metrics (ranked by importance):

  1. Numerical precision — north star, drives all decisions
  2. Citation accuracy — does it point to the right source?
  3. Faithfulness — are non-numeric claims grounded in sources?
  4. Tool selection accuracy — does it pick the right tool?
  5. Latency — how fast?
  6. Cost per query — how cheap?

Every optimization decision should be measured against: "does this improve numerical precision?" If it doesn't, it's not a priority.
