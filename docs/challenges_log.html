Challenges Log

A running record of the actual technical bottlenecks encountered while building FinSage, the root causes of system failures, and the specific engineering pivots made to resolve them.

Challenge 1: The Py4J Memory Explosion & Regex Backtracking (Week 2 & 3)

Status: Resolved
Phase: Silver-layer parsing (Extracting "Risk Factors" and "MD&A" from raw SEC HTML)

The Problem

During the transformation from the Bronze to Silver layer, the pipeline was tasked with parsing raw SEC 10-K and 10-Q filings. These files are deceptively massive—often exceeding 50MB—because companies embed huge Base64 strings for graphics and massive machine-readable XBRL code blocks at the bottom of the HTML document. When attempting to run the section-splitting logic, the Databricks Single-Node cluster would either freeze completely, spin the CPU endlessly, or crash entirely with a Java heap space Out of Memory (OOM) error.

The Root Cause

The failure was a combination of three distinct architectural traps:

  • The Py4J Serialization Bottleneck: Apache Spark runs natively on the JVM (Java Virtual Machine), but my parsing logic (using BeautifulSoup and Python re) was written in Python. To process the file, Spark had to take a 50MB string, serialize it out of the JVM, and push it through the Py4J bridge into a single Python worker process. Moving gigabytes of data across this bridge suffocated the driver.
  • Catastrophic Regex Backtracking: I initially tried to use a lazy Python regex (.*?) to strip out the massive <TYPE>GRAPHIC and <XBRL> blocks. However, evaluating a lazy dot-star regex across a 50MB string causes the Python regex engine to enter "catastrophic backtracking"—trying millions of permutations and throwing the CPU into an infinite loop.
  • DOM Tree Memory Bloat: Feeding a 50MB string of garbage HTML directly into BeautifulSoup forced Python to build a massive Document Object Model (DOM) tree in memory, which instantly exhausted the worker's RAM before the actual text extraction could even begin.

The Solution: The "Native First, Python Last" Architecture

To respect the physical limits of the cluster, I completely redesigned the pipeline to handle the heavy lifting in C++/Java before Python ever touched the data:

  • Native JVM Truncation (The Machete): Since the human-readable 10-K is always the very first <DOCUMENT> in the SEC submission, I used Spark's native expr("substring_index(raw_text, '</DOCUMENT>', 1)"). This runs instantly in Spark's native engine, cleanly severing the bottom 90% of the file (images/XBRL) and reducing a 50MB payload to a ~1MB payload.
  • Native HTML Stripping: Instead of using BeautifulSoup to strip tags, I used Spark's native regexp_replace(col("main_doc"), "<[^>]+>", " ") to wipe the HTML tags directly in the JVM.
  • Lightweight Python Delegation: I only passed the resulting flat, clean, 1MB string to the Python UDF. Python simply ran a lightweight regex to find the "Item 1A" and "Item 7" headers and slice the text.

Why This Challenge Matters Beyond Parsing

This original failure also taught two of the later architectural lessons explicitly:

  • Python-first full filing parsing fails at production scale. The cluster could not survive shipping massive raw payloads across the Py4J boundary.
  • Regex-heavy cleanup over giant filings is not production-safe. The backtracking behavior made the approach both fragile and computationally expensive.

Takeaway

Never force a Python UDF to do heavy string manipulation on massive payloads. Use Spark SQL's native functions to shrink and clean the data in the JVM first, and reserve Python strictly for complex, lightweight text analysis.

Challenge 2: Redundant Re-Execution Due to Spark Lazy Evaluation (Week 4)

Status: Resolved
Phase: Silver-layer branching (Error auditing + Successful section merge)

The Problem

After stabilizing the parsing pipeline, I observed that overall execution time was unexpectedly high despite handling only ~730 filings. The pipeline would slow down significantly during the error logging (count()) and again during the final merge step.

The Root Cause

This was caused by Spark's lazy evaluation model. The transformed dataframe (df_processed) was being reused in two separate downstream actions:

  • Counting and logging failed records (df_errors.count())
  • Filtering and merging successful records into the Silver table

Since no intermediate persistence was defined, Spark recomputed the entire transformation pipeline twice. This included:

  • Reading all Bronze files again
  • Re-running all string cleaning transformations
  • Re-executing the Python UDF for section extraction (most expensive step)

This effectively doubled compute time and cluster resource usage.

The Solution: Strategic DataFrame Caching

I introduced an explicit .cache() immediately after constructing the transformed dataframe:

df_processed.cache()

This forced Spark to:

  • Execute the full pipeline only once
  • Store the resulting dataframe in memory
  • Reuse the cached result for both error logging and merge operations

Additionally, I explicitly released memory after usage:

df_processed.unpersist()

Takeaway

In Spark, multiple actions on the same dataframe can silently trigger full recomputation. Explicit caching is critical when expensive transformations (like Python UDFs) are reused across multiple downstream operations.

Challenge 3: PDF-First Metric Extraction Was the Wrong Primary Path

Status: Resolved
Phase: Metric extraction architecture

The Failed Approach

I initially treated filing documents as the primary source for financial metrics, with the idea that document parsing would pull revenue, net income, debt, and the other target metrics directly from the filings themselves.

The Root Cause

  • Document parsing is useful for narrative sections and exploratory table work, but it is not the most deterministic production source for company-level financial metrics.
  • Labels vary by company and statement layout, so equivalent concepts appear under different names and formats.
  • The blueprint requires auditable, recomputable metrics with minimal ambiguity, which is better served by the SEC's machine-readable CompanyFacts/XBRL data.

The Solution

I changed the role of the filing documents. They remain important for Business, Risk Factors, and MD&A, but the 12 numerical metrics now come from Bronze-stored SEC CompanyFacts JSON that is normalized in Silver and aggregated in Gold.

Takeaway

Use filing documents for narrative intelligence; use XBRL-backed SEC data for hard financial numbers.

Challenge 4: Pulling the SEC API Directly into Gold Broke the Medallion Contract

Status: Resolved
Phase: Bronze/Silver/Gold architecture

The Failed Approach

Because the SEC CompanyFacts API already returns clean structured JSON, it was tempting to call the API and write those numbers straight into Gold.

The Root Cause

  • Gold became dependent on a live external system that can rate limit, go down, or change over time.
  • There was no immutable record of what the SEC actually returned at a given timestamp.
  • If I later expanded from 12 metrics to 13 or more, I would have to call the SEC again for the entire company universe instead of recomputing from owned data.

The Solution

I moved the API payload into main.finsage_bronze.xbrl_companyfacts_raw as an append-only raw snapshot table. Silver now flattens and normalizes those raw snapshots, and Gold computes metrics only from Silver.

Takeaway

Bronze is not busywork; it is the reliability, auditability, and recomputation layer that protects Gold from the live internet.

Challenge 5: SEC API Pulls Failed on "Small" Protocol Details

Status: Resolved
Phase: Bronze CompanyFacts ingestion

The Problem

CompanyFacts extraction looked correct in logic, but requests still failed intermittently for specific tickers.

The Root Cause

  • The SEC endpoint is strict about request identity. A properly formatted User-Agent is required for stable access.
  • CompanyFacts URLs require CIK values padded to exactly 10 digits. Unpadded CIKs produce avoidable request failures.

The Solution

I standardized request construction in Bronze ingestion: explicit User-Agent header, deterministic zfill(10) for CIKs, and failure logging to main.finsage_bronze.ingestion_errors for retry visibility.

Takeaway

API reliability depends on protocol correctness as much as network stability. Header and identifier formatting are part of the data contract.

Challenge 6: Section Extraction False Positives from TOC and Flattened HTML

Status: Resolved
Phase: Silver narrative extraction (filing_sections)

The Problem

Some filings produced missing or low-quality Business, Risk Factors, or MD&A sections even when those sections existed in the source documents.

The Root Cause

  • Flattening all HTML into one whitespace-heavy blob increased false matches against table-of-contents lines.
  • Simple first-match regex selection was too fragile for issuer-specific heading variations.

The Solution

I switched to a hardened extractor: preserve line breaks during Spark cleanup, anchor Item matches at line starts, score multiple start/end candidates, reject short blocks with section-specific minimum word thresholds, and log partial failures to ingestion_errors instead of crashing.

Takeaway

For SEC section parsing, structure-aware extraction beats naive regex slicing. Boundary quality is an engineering problem, not just a parsing problem.

Challenge 7: Fiscal-Year Drift and Duplicate Facts Corrupting Gold Metrics

Status: Resolved
Phase: Gold metric construction from Silver facts

The Problem

Early Gold outputs mixed incompatible facts: multiple accessions for the same ticker-year, non-USD entries, and period mismatches that could leak into annual metrics.

The Root Cause

  • CompanyFacts can emit multiple valid-looking records for the same concept and fiscal year across amendments and filing revisions.
  • Without canonical selection rules, Gold may combine facts from different accessions and time windows.

The Solution

I added deterministic ranking: strict annual/instant fit scoring, USD filtering, required-metric coverage scoring per accession, canonical accession selection per ticker-year, then best-fact selection per metric by concept priority and recency.

Takeaway

Gold must be opinionated about fact selection. Deterministic ranking logic is required to make financial outputs stable and interview-defensible.

Challenge 8: Regex Is Fundamentally the Wrong Abstraction for SEC HTML Parsing

Status: Open / Known Limitation
Phase: Silver narrative extraction (filing_sections)

The Problem

The current section extractor (03_silver_decoder.py) applies regex boundary detection against flattened HTML text. While the hardened extractor (Challenge 6) improved accuracy for clean filings, a deeper structural issue remains: regex over flattened text cannot reliably model the document structure that SEC filings actually use. Several failure modes exist today and will compound as the filing universe expands beyond the current 30 tickers.

The Root Causes (Structural Defects)

Problem Why Regex Cannot Solve It Well Example from Current Data
iXBRL fragmentation Post-2019 filings wrap text in <ix:nonNumeric>…</ix:nonNumeric>, splitting "Item 1." and "Business" across multiple spans with complex CSS positioning (position:absolute, display:none). Stripping tags leaves fragmented text runs. AMZN 10-K body: "Item" and "1." live in separate <ix:*> wrappers.
ToC vs. body disambiguation "Item 1A. Risk Factors" appears in the table of contents and as the section header. Regex + position scoring is a heuristic; the correct signal is <a href="#item1a"> (ToC link) versus the anchor target <div id="item1a"> (body). All AMZN and V filings exhibit this collision.
Heading detection requires typography, not text The SEC does not mandate the literal string "Item 1". Filers use bold or large-font headings such as "1. Our Business" or <span style="font-weight:700;font-size:14pt">Risk Factors</span> with no "Item" prefix at all. GS 10-K and some financial-services filers use this pattern.
Page-break noise Filers inject <div style="page-break-before:always"> with repeating page headers and footers mid-section. Regex can pick these up as spurious start or end anchors, splitting a single section incorrectly. V 10-K inserts a page break approximately every two pages.
HTML entity zoo &#160;, &nbsp;, &#8201; (thin space), &#8239; (narrow NBSP), &mdash;, &#8212;, &rsquo;, &#8217;, and smart quotes — every filer uses a different subset, and unresolved entities corrupt boundary pattern matching. V 10-Q: "Management&#8217;s Discussion" before entity decoding.
Multi-document SGML archives <DOCUMENT> wrappers can contain exhibits, cover pages, or XBRL schemas before the main 10-K body. substring_index(…, '</DOCUMENT>', 1) always takes the first document, which is not always the main filing. Some older filings have a cover-page iXBRL document first.
Text encoding failures decode(content, 'UTF-8') fails silently on older filings that use Latin-1 or Windows-1252 payloads, producing corrupted or truncated text without raising an exception. Not yet observed in the current 30 tickers, but is a confirmed future risk for older filings.
Character substitution ("invisible font-face" issue) Some filers embed individual glyphs — particularly "f" — as CSS-styled elements rather than literal characters. When those elements are stripped, the character disappears entirely, producing artifacts like "Sa﻿ety", "In﻿ormation", and "O﻿icers" in the extracted text. MCD 10-K, V 10-Q — the missing "f" is the clearest symptom.

Why This Was Not Caught Earlier

The current 30 tickers are large-cap issuers with relatively consistent HTML structure. The hardened extractor scores well against them because they follow a broadly predictable format. These failure modes emerge at the margins: post-2019 iXBRL adoption, non-standard filers (financial services, older registrants), and document archives where exhibits precede the main body. Expanding the ticker universe or ingesting amended filings (10-K/A) will surface these issues at scale.

The Correct Long-Term Fix

The right abstraction is a proper HTML parse tree, not regex over flattened text. The production path should be:

  1. Parse with BeautifulSoup or lxml on the already-truncated (post-</DOCUMENT>) payload — the Py4J serialization bottleneck from Challenge 1 is acceptable at ~1 MB, not at 50 MB.
  2. Decode all HTML entities using html.unescape() before any boundary matching.
  3. Identify sections using anchor IDs (<div id="item1a">) and hyperlink targets (<a href="#item1a">) rather than text pattern matching.
  4. Fall back to typography signals (bold, large font-size, font-weight CSS) when anchor-based detection fails.
  5. Detect the correct main document by inspecting <TYPE> tags within the SGML wrapper rather than assuming the first <DOCUMENT> block is the filing body.

This approach is scoped as future work. The current regex extractor is accepted as a pragmatic approximation for the current 30-ticker universe.

Takeaway

Regex over flattened HTML is a local optimum, not a production solution. For SEC filings, correct section extraction requires a proper DOM parse, entity decoding, and anchor-based boundary detection — not string pattern matching.

Challenge 9: The sec-parser Swap Left a Silent Coverage Gap and a Stale-Row Poisoning Problem

Status: Resolved
Phase: Silver section extraction (03_silver_decoder.py) — post-refactor audit

The Problem

After swapping the regex-on-flattened-HTML extractor for sec-parser (Technical Decision #10), the pipeline ran to completion and surface-level coverage looked good — 30/30 tickers present in silver.filing_sections, ~1,549 rows total. But ingestion_errors told a second, uglier story: 67 errors across 11 distinct failure patterns from the same run. Digging further surfaced two orthogonal problems that the initial refactor did not address:

  • The fallback policy was all-or-nothing. extract_sections only fell back to the legacy regex extractor when sec-parser returned zero sections. For financial-institution filers (JPM, BAC, MA, PFE, parts of V), sec-parser correctly promoted Item 1. Business to a TopSectionTitle, but its classifier missed Item 1A. Risk Factors and Item 7. MD&A — so sec-parser returned {Business}, the code accepted that as "extraction succeeded", and MD&A / Risk Factors were silently dropped for ~28 filings.
  • Silver was merging new sec-parser output on top of stale regex-era rows. silver.filing_sections is a MERGE INTO table keyed on section_id = sha256(filing_id || section_name). When sec-parser failed to emit a section that the old regex run had produced, the stale row survived. 42 of the 1,549 rows (2.7%) were regex-era leftovers — with average word counts 3× larger than the new rows (28,235 vs. 9,120 for MD&A) because the old extractor's end-anchor consistently missed "Item 8" and bled from Item 7 through to the end of the document. Feeding these into the Vector Search chunker would poison 19 MSFT 10-Q chunks, 14 MA 10-K chunks, 7 BAC chunks, and 2 LCID chunks with page-footer and next-section boilerplate.

The Root Causes (per-filer forensics)

Filer Failures Root cause (reproduced locally against downloaded full-submission.txt)
MSFT 10-Q 19 (100%) sec-parser did find the right "Item 2" anchors (4 matches), but _find_heading returned the first one — a bare TopSectionTitle("Item 2") (6 characters) placed near the top of the document. _body_between then stopped at the very next heading, which was the real TitleElement("ITEM 2. MANAGEMENT’S DISCUSSION AND ANALYSIS OF FINANCIAL CONDITION AND RESULTS OF OPERATIONS"). The "body" between the stub and the real heading was near-empty and failed the 150-word minimum. Every MSFT 10-Q lost MD&A the same way.
JPM 10-K 7 (100%) sec-parser promoted Items 1, 2, 3, 4, 5, 6 to TopSectionTitle but not Item 1A (Risk Factors) or Item 7 (MD&A). Regex fallback (on flat text) finds both cleanly — 2 hits each, one in the TOC and one at the body — but the all-or-nothing fallback never fired because Business was present.
MA 10-K 7 (100%) Same classifier-miss as JPM: TopSectionTitles are Items 1, 2, 3, 4, 5, 6; Item 1A and Item 7 are not classified as either TopSectionTitle or TitleElement. Regex fallback sees 16 hits for Item 1A and 18 for Item 7.
BAC 10-K / PFE 10-K / V 10-K / V 10-Q 24 Same TopSectionTitle-classifier-miss family as JPM and MA. Blast radius varies: BAC loses only MD&A; V loses MD&A on 3 of 6 10-Ks and 7 of 19 10-Qs where heading wording drifted year-over-year.
MCD 10-K 7 (100%) Structural, not a bug. MCD files a bare 10-K that incorporates the Business / Risk Factors / MD&A content by reference from the proxy and Part III. The primary <TYPE>10-K document contains only a one-pager cross-reference table ("Item 1 — Business — See Pages 3-5"). sec-parser sees 898 elements, 1 TopSectionTitle, 0 item-matching TitleElements; the regex fallback sees each Item exactly once — only in the TOC. The content does not exist in the primary document. No amount of parser tuning can recover it.
DDOG 10-Q (1 accession) 1 Bronze data-quality issue. The full-submission.txt for accession 0001561550-22-000026 on the volume is 560 KB vs. the expected ~30 MB — the file ends mid-<XBRL> with no closing </DOCUMENT>. The SGML-unwrap regex correctly reports "no document block found". 17 of DDOG's 18 other 10-Qs process normally.

The Solution

Three targeted fixes in 03_silver_decoder.py (staleness annotation plus two extraction-logic changes):

  1. Per-section partial-fallback merge. extract_sections now always runs sec-parser first and collects whatever it finds. Only the still-missing required sections go through the regex fallback, and the fallback's output is merged into the sec-parser result by section name — sec-parser wins on ties. This preserves sec-parser's clean entity-decoded page-footer-stripped output for every section it does classify, and only enlists the regex fallback for the specific sections it misses. Recovers MD&A / Risk Factors for MA, BAC, V, and (partially) JPM.
  2. Heading-preference rule in _find_heading. When multiple heading elements match a section regex, anchors whose text is trivially short (≤ 12 characters, e.g. the bare "Item 2" stub) are filtered out in favour of descriptive full-length headings. Falls back to the original behaviour if all matches are short, so previously-working filers are unaffected. Recovers MD&A for all 19 MSFT 10-Qs.
  3. Per-row extractor_used column on silver. extractor_used moved from a UDF-level attribute (which was observability-only and dropped before the merge) into the per-section struct, and is now written to silver.filing_sections via schema.autoMerge.enabled = true. After the next rerun, staleness is trivially queryable: WHERE extractor_used IS NULL identifies rows untouched by the new run; GROUP BY extractor_used reports exactly how much of silver came from sec-parser vs. the regex fallback. This closes the observability gap that made the original staleness diagnosis so manual.

  • Silver pollution cleared. The 42 stale regex-era rows are overwritten by the next run's MERGE (same section_id, whenMatchedUpdateAll()) — because sec-parser or the regex fallback now produces those same (ticker, form, section) tuples cleanly.
  • Error count drops from 67 to ~15. Residual errors are MCD 10-K ×7 (structural, accepted), PFE 10-K MD&A ×7 and JPM 10-K MD&A ×7 (flat-text candidate below the 400-word fallback threshold), plus ~3 long-tail (DDOG truncated file, JPM 10-Q single edge case, LCID edge case).
  • Observability closed. SELECT extractor_used, COUNT(*) FROM main.finsage_silver.filing_sections GROUP BY 1 will return a clean three-way split: sec-parser dominant, regex-fallback small tail, NULL = nothing (or a small set of rows the next rerun genuinely did not touch).

Known Residuals and Deferred Work

  • MCD 10-K (7 filings): downgrade the error to a warning tier, or add a tier-3 extractor that pulls content from EX-99 annual-report exhibits. Deferred; MCD simply does not file its narrative content in the 10-K primary document.
  • JPM 10-K and PFE 10-K MD&A: regex fallback finds the Item 7 anchor but the candidate text between Item 7 and Item 8 falls below the 400-word minimum after tag stripping, almost certainly because these filers split MD&A into a TOC-page skeleton and a separately-sectioned financial review later in the document. Candidate approach: a tier-3 extractor that concatenates all flat-text fragments between the first item 7 and the first item 8 anchors regardless of min-words, or a bank-filer-specific heuristic. Tracked as follow-up.
  • DDOG 2022-Q1 10-Q truncation: bronze-layer re-ingest for that single accession. Worth a bronze-wide scan for other full-submission.txt files smaller than, say, 1 MB that might be silently truncated.

Takeaway

"Pipeline ran to completion" is not the same as "extraction succeeded." When a merge-based silver layer swaps extractors, staleness auditing is as important as the new extractor's accuracy — because the old extractor's wins will survive the new extractor's losses, and silently. Per-row provenance on silver (extractor_used), per-section partial-fallback merging, and a local-reproducer harness for every ticker-specific failure mode are the three ingredients that turn an opaque "30/30 tickers" coverage number into a trustworthy one.

Challenge 10: The Chunker Orphan Bug — When a Content-Hashed Merge Key Silently Doubles the Table

Phase: Vector chunking (05_vector_chunker.py) — post-silver-refactor audit

The Problem

After the Challenge 9 silver fixes landed (commit 3c6c051) and notebook 05 was rerun, surface-level checks on silver looked correct — 30 tickers, 1,572 rows, a clean sec-parser + regex-fallback mix. But when I cross-checked the downstream chunks table, main.finsage_gold.filing_section_chunks had 105,029 rows, not the ~45k that silver's new content should have produced. Per-ticker spot checks confirmed the gap: BAC had 8,472 chunks, GS 10,093, NET 8,013 — roughly double what the silver row counts could justify. A GROUP BY (filing_id, section_name, chunk_index) on AAPL-10-K-2022 returned multiple rows at the same positional index, each with a different chunk_id. Two generations of chunks were coexisting in the table. The Vector Search index was about to embed both.

Why It Happened

deterministic_chunk_id in 05_vector_chunker.py hashes over five inputs: 

chunk_id = sha256(filing_id || section_name || chunk_index || chunk_text || chunk_version)

Including chunk_text in the hash was originally a defensive choice — any change in chunk contents would produce a new ID, so downstream consumers could detect drift by ID alone. But coupled with the chunker's write pattern — 

target.alias("t")
    .merge(df_chunks.alias("s"), "t.chunk_id = s.chunk_id")
    .whenMatchedUpdateAll()
    .whenNotMatchedInsertAll()
    .execute()

— it meant that any upstream text change broke the merge key. When sec-parser replaced the regex extractor, the same logical section (same filing_id, same section_name, same chunk_index) produced different chunk_text (cleaner, no HTML entities, no page-footer bleed), and therefore a completely different chunk_id. The MERGE found zero t.chunk_id = s.chunk_id matches, executed 45,136 inserts, and left all 59,893 regex-era rows untouched as orphans. The DESCRIBE HISTORY evidence was unambiguous: 

version 21 | MERGE | numTargetRowsInserted=45136 | numTargetRowsUpdated=0 | numTargetRowsDeleted=0

Zero updates and zero deletes across a 45k-row "upsert" is the diagnostic signature of a content-hashed key whose content has changed upstream. The per-run_id breakdown confirmed the same story from the row-provenance side:

run_idchunked_atrowsprovenance
aa5609e0-…2026-04-2159,893regex-era chunks (pre-Challenge 9 fix)
b7c716da-…2026-04-23 AM45,136sec-parser-era chunks — new IDs, stacked on top

A secondary issue surfaced during the post-fix deploy validation. The polling loop in notebook 06's final cell exited the moment ep.state.ready == "READY", without checking ep.state.config_update or whether the version just deployed was actually in ep.config.served_entities. Databricks serving keeps the previous version marked READY throughout a rollout (it's still serving traffic until the atomic traffic swap), so the poll exited instantly and the notebook's post-deploy live tests ran against the old version. Easy to miss — the tests returned plausible answers from a working endpoint.

The Solution

Two commits on dev:

  1. 8e1d478 — fix(chunker): delete prior chunks per (filing_id, section_name) before re-append. Replaces the single MERGE-on-chunk_id write with a two-step scoped replace. The source is df_chunks.select("filing_id", "section_name").distinct(), which keeps the MERGE 1-to-many safe: 
    affected_pairs = df_chunks.select("filing_id", "section_name").distinct()
target.alias("t")
    .merge(affected_pairs.alias("p"),
           "t.filing_id = p.filing_id AND t.section_name = p.section_name")
    .whenMatchedDelete()
    .execute()
df_chunks.write.format("delta").mode("append").saveAsTable(TARGET_TABLE)
    Every target row whose (filing_id, section_name) pair is being rewritten gets deleted first; the fresh chunks land clean. Idempotent per section regardless of upstream text changes. Ticker-filtered runs stay safe — the DELETE only touches pairs actually present in the current source.
  2. 8672b08 — fix(06): poll until config_update=NOT_UPDATING and new version is in served_entities. Three-way exit guard on the deploy-wait loop: state.ready == READY, state.config_update == NOT_UPDATING, and str(model_version) in {se.entity_version for se in ep.config.served_entities}. Post-deploy live tests now provably hit the version the notebook just deployed, not the previous one. Also drops a dead precedence-bug branch on the old loop ("FAILED" in state.upper() or "NOT_READY" in state.upper() and … — and binds tighter than or, shadowing the intended FAILED-exit with a no-op).

Validation

After pulling 8e1d478 and rerunning notebook 05, the chunks table's DESCRIBE HISTORY confirmed the intended shape of the write: 

version 23 | MERGE | numTargetRowsDeleted=105029   (scoped delete wiped both generations)
version 24 | WRITE | 45136 rows appended            (single fresh generation)

Downstream, a three-probe end-to-end audit against the redeployed agent endpoint validated every tool path:

This was impossible before Challenge 9 — AMZN had zero chunks in the index.

Known Residuals and Deferred Work

  • Silver-delete edge case. The scoped-delete fix covers sections that appear in the current source. If a filing is ever removed from silver (e.g. silver rows are deleted or a ticker is retired), the chunker will leave its chunks behind — it only touches (filing_id, section_name) pairs that are in the current source. Silver only grows today, so this is not an active problem; flagging in case silver ever gains delete semantics.
  • Agent scoping drift between annual and quarterly retrieval. The SYSTEM_PROMPT rule for "most recent" questions instructs the agent to call a metrics tool first to find the latest year, then pass that year to search_filings. In practice, the agent uses the annual metrics table's latest year (e.g. FY2024 for V) even for 10-Q retrieval, but silver tags 10-Qs filed in 2025–2026 as FY2025/FY2026. On the V 10-Q regulatory-risk probe, all four retrieval attempts (with fiscal_year=2024) returned below the similarity threshold; the agent fell back gracefully. A prompt-tuning target for the V0 evaluation phase — the fix is to scope latest-year resolution by filing_type (annual → company_metrics, interim → company_metrics_quarterly), or to drop the year filter on "most recent" questions entirely and let retrieval rank.

Takeaway

An idempotent MERGE is only idempotent within the ID invariants it enforces. When the key hashes over content, upstream content changes silently violate idempotency — not by raising an error, but by treating updated rows as new inserts and orphaning the old ones. The only safe upsert against a content-hashed key is to scope the replace to a key that is actually stable across the change (here, (filing_id, section_name)), not the hash itself. And "endpoint READY" is a necessary, not sufficient, signal for post-deploy validation: always require that the rollout has actually landed the version you intended to test.

Challenge 11: The v21 Eval "97 Failures" Were Mostly the Scorers and the Tool Output, Not the Agent

Phase: Agent evaluation (07_evaluation.py + src/evaluation/scorers.py) — first run of the v21 LLM-driven agent against the 100-question stratified ground-truth set.

The Problem

v21 was the first agent build with the deterministic-regex shortcut removed (commit 72dbe85): every question now routes through the LLM tool loop with the new 170-line SYSTEM_PROMPT. The first 100-question eval run persisted to main.finsage_gold.eval_question_outcomes with a sum-of-fails count of 97 across 8 scorers, on top of a v20 baseline that had only ~35 sum-of-fails. The headline scorers all moved the wrong direction:

On the surface this looked like the new SYSTEM_PROMPT had broken the agent comprehensively. Drilling into eval_question_outcomes by scorer × outcome, then sampling the rationale + agent_response + expected_response for each fail, told a different story. Three independent root causes were responsible for the bulk of the regression — none of them were the agent producing wrong content.

Root Cause 1 — The Metrics Tool's Formatted Output Hid total_liabilities, total_equity, and rd_expense

Drill-down on the correctness failures showed a recurring pattern across A001 (PLTR FY25 total_equity), A011 (SNOW FY24 total_liabilities), A014 (BAC FY22 total_liabilities), A019 (SBUX FY24 total_equity), A021 (JNJ FY22 total_equity), A023 (PFE FY25 total_liabilities), A028 (DDOG FY22 total_equity), and B012 (PLTR Q1 FY24 total_liabilities). The agent's response shape was identical in every case: 

"Bank of America Corporation's total liabilities in fiscal year 2022 were
not directly provided. However, we can see that the total assets were
$3051.38B and the total debt was $275.98B. To find the total liabilities,
we would need more information…"

That phrasing — "not directly provided", followed by an attempt to derive — is the agent following the SYSTEM_PROMPT correctly: never invent data. The bug was upstream: get_company_metrics and get_quarterly_metrics in 06_rag_agent.py formatted the tool output with these lines and these lines only: 

FY2022:
  Revenue:              $94.95B
  Net Income:           $27.53B
  Gross Profit:         N/A
  Operating Income:     $30.97B
  Operating Cash Flow:  $108.70B
  Total Assets:         $3051.38B
  Total Debt:           $275.98B          ← total_debt shown
  Gross Margin:         N/A
  Revenue YoY Growth:   −5.7%
  Debt/Equity:          1.01x
  Data Quality Score:   90%

Notice the absence of Total Liabilities, Total Equity, and R&D Expense. The cache holds all three (the loader's SELECT in _load_metrics_cache pulls them, and a direct cache probe confirmed BAC FY2022 total_liabilities = $2,778,178,000,000 — the exact eval target to the dollar). They just weren't in the _fmt output. The agent saw "total_assets" and "total_debt" in its tool result and concluded — correctly given its view — that total_liabilities and total_equity weren't available.

Why this had been masked in v20: the deterministic-regex shortcut in v20 short-circuited many of these questions to a pre-canned answer that bypassed the tool entirely. v21's LLM-driven loop is the first time the LLM has ever read the formatted tool output for these question shapes. The cosmetic gap had been there for weeks.

Cascading impact: this single missing block of three lines in the tool output produced ~12 sum-of-fails across correctness, numerical_tolerance, and cites_ticker_and_year. Fixing it (six new f"\n Total Liabilities: …" lines across the two tools) clears all three fail counts in one change.

Root Cause 2 — The Citation-Format Safety Net Corrupted Refusals

F004 (refusal_test on MCD 10-K text) came back with the response shape: 

"[SUMMARY] Unfortunately, I couldn't find any relevant narrative discussion
of FY2023 from McDonald's (MCD) 10-K filing using the search query…"

The leading [SUMMARY] tag is wrong on a refusal — refusals don't quote retrieved text, so the verbatim/summary contract doesn't apply. It got prepended by _enforce_citation_format, the final-stage safety net inside predict() that adds the tag if the LLM forgot to include one on a retrieval-grounded answer: 

if not used_retrieval:
    return content
final = content
if not re.search(r"\[(VERBATIM|SUMMARY)\]", final, …):
    final = "[SUMMARY] " + final
…

For F004, the agent did call search_filings (so used_retrieval=True), but the search returned zero passages because MCD's 10-K narrative is the known section-extraction gap (Challenge 9 footnote — silver captured 0 sections from MCD 10-Ks). The LLM correctly produced a refusal, but the safety net saw used_retrieval=True and prepended [SUMMARY] regardless of whether the search actually returned anything. The refusal got tagged as a summary of nothing.

Fix: gate the safety net on tool_sources being non-empty (i.e., search returned at least one extractable [Source: ...] line) rather than on used_retrieval=True. Plus a new explicit MCD-10K-gap refusal worked example in the SYSTEM_PROMPT so the LLM produces the right shape from the start. Cascading impact: closes 1 sum-of-fail directly and unlocks refusal_correctness from rejecting F004 due to the corrupted prefix.

Root Cause 3 — Two Scorers Were Marking Correct Agent Behavior as Failure

(a) citation_format required [VERBATIM]/[SUMMARY] tags on filing-metadata answers

The 10/10 → 0/10 swing on citation_format was entirely the E-category (filing_metadata) rows. The agent in v21 correctly routes those questions to the new get_filing_metadata tool and produces clean, terse answers like: 

"Microsoft Corporation filed its FY2021 10-K with the SEC on July 29, 2021.

[Source: MSFT | FY2021 | 10-K Cover Page]"

Per the SYSTEM_PROMPT, filing-metadata answers do not get a [VERBATIM]/[SUMMARY] tag — those are for retrieval-grounded answers that quote filing text. The scorer's category gate let E-category through without checking that source_section was "10-K Cover Page" (the tell that this is a deterministic lookup, not a retrieval). It then dutifully marked all 10 as FAIL because the tag was absent.

Why this was 10/10 PASS in v20: v20's deterministic-shortcut path didn't have a get_filing_metadata tool — those questions either fell through to search_filings (which produces VERBATIM/SUMMARY-grammar answers) or got served by the canned-regex path which the scorer's category set happened to whitelist. v21 routing them through the proper deterministic-lookup tool is the correct shape; the scorer hadn't caught up.

Fix: SKIP the scorer when source_section is in {"metrics", "10-K Cover Page", ""}, OR when the response itself shows a [Source: ... | metrics] or [Source: ... | Cover Page] line. Cascading impact: 10 sum-of-fails turn into 10 SKIPs (correctly inapplicable).

(b) refusal_correctness's phrase whitelist didn't know modern conversational refusals

The 5/5 → 0/5 swing on refusal_correctness had a different mechanism. The scorer requires two things: (1) the response contains a refusal phrase from _REFUSAL_PHRASES, and (2) the response mentions at least one expected reason token from _REFUSAL_CONTEXT_TOKENS[qid]. The expected_tokens check passed on all 5 (the agent's responses correctly named "2030", "IBM/corpus", "Q4", "MCD/10-K", "FB/Meta"). What failed was the refusal-phrase check, because the whitelist only had explicit "I cannot/can't/decline" verbs: 

_REFUSAL_PHRASES = (
    "cannot", "can't", "cant", "unable", "not available", "not present",
    "not in the corpus", "out of scope", "outside", "no data", "no record",
    "decline", "i'm sorry", …
)

The new LLM-driven agent, with its conversational-tone instructions, refuses in the modern shape:

  • F001 (AAPL FY2030): "Apple's fiscal year 2030 hasn't occurred yet — it's a future period…"
  • F002 (IBM): "IBM isn't in the FinSage corpus…"
  • F003 (MSFT Q4): "Q4 isn't stored as a standalone quarter…"
  • F004 (MCD 10-K text): "Unfortunately, I couldn't find any relevant narrative…"
  • F005 (FB/GOOG): "FB was Meta's former ticker… isn't currently in…"

Every one of these is a refusal. None of them tripped the whitelist, because the whitelist didn't have "isn't", "hasn't", "couldn't", "former ticker", "future period". Five clean refusals scored as five FAILs.

Fix: broaden _REFUSAL_PHRASES to cover the negation+auxiliary forms ("isn't", "is not", "aren't", "wasn't", "hasn't", "haven't", "doesn't", "didn't", "couldn't", "could not", "don't have"), period-related hooks ("future period", "hasn't occurred", "yet to be filed"), and ticker hooks ("former ticker", "deprecated", "no longer in"). Three new regression unit tests added (test_refusal_correctness_recognizes_modern_phrasings covers all 5 F-cases). Cascading impact: 5 sum-of-fails turn into 5 PASSes.

Why I Initially Thought the Agent Was Broken

The aggregate fail counts on the run summary make the regression look like a content failure. The "ground-truth answer is right, agent's answer is wrong" reading is the natural first interpretation, and it's wrong here in two ways:

  • Sum-of-fails is not a count of unique broken questions. Question A014 (BAC FY22 total_liabilities) tripped correctness + cites_ticker_and_year + numerical_tolerance simultaneously — three sum-of-fails from one root cause. The 97 sum-of-fails in the v21 run came from approximately 25–30 unique questions, each multi-tripping. Fixing one root cause clears the whole stack.
  • Multi-dimensional scoring catches what a single judge misses. The single LLM-judge would have nudged a fuzzy "correctness" score by some amount and given no diagnostic signal. The eight independent scorers tell you exactly which dimension regressed — and three of the regressions (citation_format 10/10 → 0/10, refusal_correctness 5/5 → 0/5, retrieval_grounded_when_used errors) had the same diagnostic shape: the scorer doesn't know about a tool/shape that didn't exist in v20. That's exactly the regression class you can't catch with a single fuzzy judge, because there's nothing for it to anchor on.

The Solution

Takeaway

When an eval regression looks comprehensive — 97 sum-of-fails, every scorer worse — the first move is not to assume the agent broke. It's to query eval_question_outcomes and read the per-scorer rationale on a sample of the FAIL rows. In v21's case, three of the largest single-scorer regressions (citation_format 10/10 → 0/10, refusal_correctness 5/5 → 0/5, the F004 [SUMMARY] tag corruption) had identical diagnostic shape: the agent's behavior was correct, and the scorer or the tool output didn't know about a code path that v20 didn't have. The remaining regression (correctness 87 → 63) was traced to a single missing block of formatted-output lines in get_company_metrics hiding total_liabilities / total_equity / rd_expense from the LLM. One commit fixes all four. The lesson: multi-dimensional scoring catches what a single LLM judge masks, but only if you read the per-scorer rationale before reaching for "the model regressed".
