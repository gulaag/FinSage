"So let me start with how the entire project is deployed and orchestrated. FinSage uses a Databricks Asset Bundle — this file is the single source of truth for the entire job topology. Clusters, tasks, schedules, environment promotion — all defined as code."

🔍 Code Callout — variables: block (line ~36)

"Notice the variables block at the top. I parameterize everything — the catalog name, environment label, cluster version, even the ticker filter. These get injected into every notebook via base_parameters, and the notebook reads them with dbutils.widgets.get(). This means the exact same notebook code runs correctly in both dev and prod — zero code changes between environments."

🔍 Code Callout — job_clusters: → finsage_cluster (line ~96)

"I define one shared job cluster and all five tasks reference it by key. The alternative — a new cluster per task — would cost five separate cold-start penalties, around five to eight minutes each. By sharing one ephemeral cluster across the whole DAG, I cut that down to a single startup cost."

✅ Key Point — dev vs prod targets

"The targets block defines two environments. The dev target has the schedule paused by default so I don't burn compute credits during development. The prod target unphrases the schedule and is the only target CI/CD ever touches. Promotion is just a CLI flag — databricks bundle deploy -t prod."

CI/CD · Authentication

GitHub Actions Pipeline — .github/workflows/deploy.yml

click to mark done

⚡ Action

Open .github/workflows/deploy.yml — point to the three jobs: unit-tests, bundle-validate, deploy-prod

💬 Say This

"Every push to main triggers a three-job sequential pipeline. First, pytest runs on GitHub's runner — no Databricks connection needed, no cloud cost, and it completes in seconds. If tests pass, bundle validate checks the YAML syntax against the live workspace API to catch config errors before any deployment. Only if both gates pass does deploy-prod run."

🔍 Why This Matters

"The pytest gate is critical. If my TARGET_CONCEPT_MAP ever gets corrupted — say someone maps Revenues to the wrong metric — the unit tests catch that in CI and block the deploy entirely. Bad code never reaches the warehouse."

⚠️ Important — Be Ready For This Question: "Why isn't run_as configured?"

"Great question — and I'm glad you noticed. My CI/CD pipeline is fully architected for Machine-to-Machine OAuth via a Service Principal. The Databricks CLI is configured to read DATABRICKS_HOST, DATABRICKS_CLIENT_ID, and DATABRICKS_CLIENT_SECRET from GitHub secrets, call the workspace OIDC token endpoint, and receive a short-lived bearer token. No PAT is used anywhere."

Continue with:

"However — I intentionally commented out the run_as block in databricks.yml. I don't currently have Workspace Admin privileges to create and register the Service Principal in this specific environment. Rather than let CI fail on every deploy, I treated this as a deliberate 'Ops Handoff' — the configuration is documented, the pattern is correct, and it's a two-minute fix once an admin provisions the SP. For this demo, I authenticate using User-to-Machine OAuth via databricks auth login. I designed around the constraint rather than blocking myself."

Infrastructure as Code

Terraform — terraform/main.tf

click to mark done

⚡ Action

Open terraform/main.tf — point to databricks_cluster_policy, databricks_secret_scope, and the SP data source

💬 Say This

"I use Terraform for a specific reason — there's a separation of concerns between infrastructure and orchestration. Terraform manages the long-lived workspace state that outlasts any single deploy. The DAB manages the daily job definition that's redeployed on every push. They're complementary, not competing."

🔍 Three Things Terraform Owns Here

"First — the cluster policy. It enforces hard limits: maximum four workers, thirty-minute auto-termination, and mandatory cost tags on every cluster. No engineer can accidentally spin up an uncapped cluster and rack up a surprise bill."

"Second — the secret scope. Runtime credentials never live in code. They're stored in the Databricks secret scope and notebooks read them with dbutils.secrets.get()."

"Third — the Service Principal data source. Terraform resolves the SP application ID so it can be referenced in the DAB without hardcoding any credential."

Notebook 01

Schema Setup — 01_schema_setup.py

click to mark done

⚡ Action

Open databricks/notebooks/01_schema_setup.py — scroll to the %sql CREATE SCHEMA IF NOT EXISTS block and the CREATE TABLE IF NOT EXISTS DDL

💬 Say This

"Notebook one is intentionally boring — that's by design. It's pure DDL. It creates the three Medallion schemas — finsage_bronze, finsage_silver, and finsage_gold — inside Unity Catalog."

🔍 Idempotency — IF NOT EXISTS

"Every single DDL statement uses IF NOT EXISTS. This notebook is idempotent by definition. I can re-run it on an existing prod environment after a schema drift incident and it will not corrupt or truncate anything. It's also the first task in the DAB DAG, so the full pipeline can't start on a broken schema."

✅ Key Point — Unity Catalog binding

"Notice the CATALOG variable at the top. The schema names are not hardcoded — they're bound to whatever catalog the DAB injects. In dev I might point this at a personal sandbox catalog; in prod it hits main. The notebook doesn't care."

Notebook 02 · Bronze Layer

Auto Loader Ingestion — 02_bronze_autoloader.py

click to mark done

⚡ Action

Open databricks/notebooks/02_bronze_autoloader.py — point to the RESET_PIPELINE flag and the cloudFiles stream config

💬 Say This

"The Bronze layer has one rule: append-only, zero business logic. Whatever comes from the SEC API goes straight in. This gives me a complete audit trail — if Silver or Gold ever produce wrong numbers, I can always reprocess from Bronze without hitting the SEC API again."

🔍 Auto Loader — Exactly-Once Ingestion

"I use Databricks Auto Loader with the cloudFiles format to stream raw SEC filing bytes. The key guarantee here is exactly-once delivery, backed by a RocksDB checkpoint. If the job crashes mid-run, the next run resumes from exactly where it stopped — no duplicate files, no missed files."

🔍 availableNow=True trigger

"I use availableNow=True as the trigger, which makes the stream behave like a batch. It processes all new files since the last checkpoint, then stops. This is intentional — I don't want a 24/7 streaming cluster running for a nightly batch job."

✅ Safety Valve — RESET_PIPELINE

"Notice the RESET_PIPELINE = False flag at the top. This is a safety valve for full re-ingestion. It's off by default and requires a deliberate code change to activate — you can't accidentally nuke your Bronze tables by re-running the job."

Notebook 03 · Silver Layer — ⚠️ Hardest Layer

Silver Decoder — 03_silver_decoder.py

click to mark done

⚡ Action

Open databricks/notebooks/03_silver_decoder.py — point to flatten_companyfacts function, TARGET_CONCEPT_MAP, and the split_udf / chunk_sections_udf

💬 Say This

"This was the hardest notebook to build. Silver does two completely different jobs. Part A flattens the XBRL JSON. Part B parses the raw 10-K HTML into three readable sections. Let me walk through each."

🔍 Part A — XBRL Flattening + TARGET_CONCEPT_MAP

"The flatten_companyfacts function is a pure Python generator that I run with flatMap over the Bronze RDD. I intentionally kept it off Spark's columnar engine because the XBRL JSON structure is deeply nested and irregular — Python handles that far more cleanly than trying to unpack it with Spark SQL."

"The key to normalisation is TARGET_CONCEPT_MAP. Every company files under slightly different XBRL tags — Apple uses RevenueFromContractWithCustomer, JPMorgan uses RevenuesNetOfInterestExpense. My map collapses 30-plus raw XBRL concepts into 11 clean metric names. This is the single source of truth, it lives in src/finsage/constants.py, and it's independently unit-tested in CI without needing a Spark session."

🔍 Part B — HTML Section Extraction via UDF

"Part B is the more complex one. SEC 10-K filings are enormous HTML files — some are several megabytes. I need to extract three specific sections: Item 1 Business, Item 1A Risk Factors, and Item 7 MD&A."

"I strip the HTML in stages using regexp_replace — first removing base64 images, then scripts and styles, then replacing block-level tags with newlines, and finally stripping all remaining tags. Then the split_udf uses regex boundary detection to find section headers, scores candidate matches by word count, and picks the best block. This runs as a Spark UDF vectorised across all executors in parallel."

✅ Key Point — SHA-256 Dedup + MERGE

"Every financial statement record gets a SHA-256 statement_id derived from the ticker, fiscal year, accession number, and metric name. The Silver write is a Delta MERGE on that hash. Re-running this notebook a hundred times produces the same Silver table — no duplicates."

Notebook 04 · Gold Layer

Gold Metrics — 04_gold_metrics.py

click to mark done

⚡ Action

Open databricks/notebooks/04_gold_metrics.py — scroll to the concept_priority Window function, then to the DeltaTable.forName(...).merge() statement near the bottom

💬 Say This

"Gold is where the raw metrics become analyst-ready KPIs. The output is one wide row per ticker per fiscal year with sixteen derived metrics — gross margin percentage, revenue year-over-year growth, debt-to-equity ratio, and a data quality score that tells downstream consumers how complete that row is."

🔍 The Hard Problem — Canonical Accession Selection

"This was a subtle data quality challenge. A company can amend its 10-K multiple times, producing multiple accession numbers for the same fiscal year. I can't just take the latest one — the amendment might have only corrected a footnote and might be missing metrics the original had. So I use a Window function with concept_priority to rank accession numbers by how many of the nine core metrics they contain, then pick the best-coverage accession before any aggregation."

🔍 Delta MERGE — the Upsert Pattern

"The write to Gold is a Delta MERGE — a true UPSERT. I match on ticker and fiscal_year. If the row already exists, I update it. If it's new, I insert it. This means the Gold table is always current without ever having duplicate rows, and it's safe to re-run after any upstream fix."

✅ Key Point — Fiscal Year Alignment Guard

"I apply a strict 350 to 380 day duration filter on flow metrics like revenue and cash flow. This prevents quarterly or stub-period facts from polluting the annual Gold table. A revenue figure for a nine-month period would look like a thirty percent drop in a trend chart and confuse every analyst downstream."

Notebook 05 · Vector Layer — 🤖 Data Engineering → AI Bridge

Vector Chunker — 05_vector_chunker.py

click to mark done

⚡ Action

Open databricks/notebooks/05_vector_chunker.py — point to the CHUNK_TOKENS = 512 config, the deterministic_chunk_id function, and the @F.pandas_udf decorator on chunk_sections_udf

💬 Say This

"This is the notebook that bridges Data Engineering and AI. It takes the clean narrative sections from Silver — the Business description, Risk Factors, and MD&A — and converts them into a format that a Retrieval-Augmented Generation system can query."

🔍 Tiktoken — Token-Aware Chunking

"I use OpenAI's Tiktoken library to split text into chunks of exactly 512 tokens with a 64-token overlap between adjacent chunks. The overlap is critical — it ensures that a sentence split across a chunk boundary doesn't get its context destroyed. Using tokens rather than characters is important because the embedding model — text-embedding-3-large — operates on tokens, and I don't want to silently truncate chunks that exceed the context window."

🔍 deterministic_chunk_id — Idempotent Vector Writes

"Every chunk gets a SHA-256 chunk_id derived from the filing ID, section name, chunk index, chunk text, and chunk version string. This is deterministic — the same input always produces the same ID. The write to Gold is again a Delta MERGE on this hash, so re-running the chunker never creates duplicate vectors in the index."

🔍 Pandas UDF — Vectorised Chunking at Scale

"The chunking runs inside a Pandas UDF — notice the @F.pandas_udf decorator. Unlike a regular Python UDF which processes one row at a time with high serialisation overhead, a Pandas UDF receives and returns entire Arrow-backed batches. Tiktoken and all the chunking logic runs in pure Python locally on each executor, and Spark never has to touch the tokenizer internals."

✅ Key Point — Databricks Vector Search

"The final step provisions a Databricks Vector Search Delta Sync index backed by the databricks-bge-large-en embedding model. Because I enabled delta.enableChangeDataFeed on the source table, the index automatically syncs whenever new chunks are merged in — no manual re-index step required."
