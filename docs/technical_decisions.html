Technical Decisions

This document captures the strategic "why" behind the FinSage architecture, specifically outlining why certain Medallion and Agent design choices were made over alternative approaches.

1. Scope: 30 Companies (Depth Over Breadth)

Decision: The platform is explicitly constrained to 30 Tier-1 companies (yielding ~730 filings total), rather than attempting to scrape the entire Russell 3000 or S&P 500.

The Engineering Rationale:

  • Deterministic Validation: The core of this project is proving numerical precision and extraction accuracy (the "North Star Metric"). With 30 companies, I can manually open an SEC PDF, look at the Balance Sheet, and verify that my pipeline extracted the exact same number. You cannot manually audit an automated pipeline scraping 3,000 companies, which leads to silent failures and "hallucinated" downstream analytics.
  • Evaluation Rigor: I am building a 100-question evaluation suite to test the RAG Agent. Ensuring those 100 questions have perfectly accurate ground-truth answers requires a tightly controlled dataset.
  • Infrastructure Agility: Processing 30 companies allows me to iterate on pipeline logic, test new chunking strategies, and run the entire Medallion architecture on a cost-effective Databricks cluster in minutes.
  • Scalability is a Config Change: Because the foundation is built on Auto Loader, Unity Catalog, and Delta Lake, the architecture is already inherently distributed. Scaling to 300 companies later requires zero code rewrites—it only requires increasing the cluster size and updating the target ticker list.

2. Decoupled Parsing: "Native JVM First, Python Last"

Decision: Perform all heavy document truncation and initial cleaning using Spark's native SQL functions, and restrict Python UDFs to lightweight section slicing.

The Engineering Rationale:

Initially, I attempted to pass the raw, 50MB binary content from the Bronze layer directly into a Python UDF to be parsed by BeautifulSoup. This caused massive Py4J serialization overhead and crashed the worker nodes.

By decoupling the work, I leverage the exact strengths of the Databricks engine:

  • Spark SQL (JVM): Handles the brute-force work. substring_index and regexp_replace slice off the massive XBRL/Image data dumps and strip HTML tags instantly across distributed nodes.
  • Python UDF: Handles the surgical work. It receives a tiny, 1MB string and uses Python's robust re library to identify "Item 1A" and "Item 7" section boundaries.

This architectural pivot reduced processing time per document from minutes (crashing) to milliseconds (stable).

3. Error Auditing: The "Never Swallow Failures" Constraint

Decision: The text-extraction UDF must never crash the pipeline or silently drop records when it encounters a malformed SEC filing. It must route bad data to a dedicated audit table.

The Engineering Rationale:

In real-world data engineering, source data is chaotic. Older SEC filings, amended 8-Ks, or filings from specific sectors often do not contain standard "Item 1A" or "Item 7" string headers.

Instead of writing fragile try/except blocks that just skip the file, I designed the UDF to return a complex StructType containing both the sections array and an error string flag.

  • If a document parses perfectly, the error is null, and the sections are exploded and merged idempotently into the Silver table.
  • If a document fails, the pipeline catches the error string, attaches the file_path and a failed_at timestamp, and writes it directly to main.finsage_bronze.ingestion_errors.

This pattern guarantees high pipeline uptime while giving me a queryable "to-do list" of edge cases to investigate and fix in future sprint cycles, perfectly adhering to the Blueprint 4.1 compliance standards.

4. Performance Optimization: Strategic Use of Caching

Decision: Explicitly cache intermediate DataFrames when reused across multiple Spark actions.

The Engineering Rationale:

Spark uses lazy evaluation, meaning transformations are not executed until an action (like count() or write()) is triggered. In the Silver layer pipeline, the transformed dataframe (df_processed) was used in two independent branches:

  • Error auditing (counting and logging failed records)
  • Processing and merging successful records

Without caching, Spark recomputed the entire pipeline for each branch. This included re-running expensive operations such as:

  • Regex-based text cleaning
  • Python UDF execution for section extraction

To eliminate this redundancy, I explicitly cached the dataframe:

df_processed.cache()

This ensured:

  • The transformation pipeline executes only once
  • Intermediate results are stored in memory
  • All downstream actions reuse the cached data

Finally, I used:

df_processed.unpersist()

to release memory after all operations completed, preventing unnecessary memory pressure on the cluster.

Impact

  • Reduced total pipeline execution time by ~50%
  • Eliminated redundant UDF execution
  • Improved cluster efficiency and stability

Takeaway

Caching is not an optimization—it's a necessity when reusing expensive transformations in Spark pipelines.

5. Bronze-First SEC API Snapshotting

Decision: Never pull the SEC CompanyFacts API directly into Gold. Always land the raw JSON payload in Bronze first.

The Engineering Rationale:

  • Operational Resilience: Public APIs rate limit, return intermittent failures, and sometimes become temporarily unavailable. If Gold depends on a live SEC call, Gold becomes fragile for reasons unrelated to my business logic.
  • Auditability: By storing the exact raw JSON payload plus fetched_at, I preserve a time-stamped record of what the SEC actually returned at ingest time.
  • Recomputation: If I add a 13th metric later, I can recompute Silver and Gold from stored Bronze snapshots instead of re-querying the SEC for the full company universe.

What This Looks Like in FinSage

The CompanyFacts API is ingested into main.finsage_bronze.xbrl_companyfacts_raw as immutable snapshots. Silver flattens those payloads, and Gold reads only from Silver. This is the correct Medallion separation of concerns.

Takeaway

Bronze is the reliability shield between FinSage and the live internet.

6. XBRL as the Numerical System of Record

Decision: Filing documents remain valuable for qualitative sections, but hard financial metrics must come from machine-readable SEC CompanyFacts/XBRL facts.

The Engineering Rationale:

  • Deterministic Accuracy: Structured SEC facts eliminate the ambiguity that comes from trying to visually parse financial tables across many issuers and formats.
  • Precision Over Flash: A project that claims interview-grade financial accuracy should use the most deterministic available source for numbers, not the most visually impressive extraction path.
  • Clean Layer Responsibilities: Filing documents are still excellent for Business, Risk Factors, and MD&A, but Gold metrics should not depend on narrative documents when a structured source exists.

What This Looks Like in FinSage

main.finsage_silver.financial_statements is built from normalized CompanyFacts concepts, while main.finsage_silver.filing_sections stores the narrative sections needed for retrieval and analysis.

Takeaway

Use filing documents for qualitative context and XBRL-backed SEC data for exact numerical truth.

7. Silver Normalization Before Gold Aggregation

Decision: Gold should never have to understand SEC concept sprawl or source-system naming variation.

The Engineering Rationale:

SEC CompanyFacts exposes multiple synonymous concepts for the same business idea, such as Revenues, SalesRevenueNet, and RevenueFromContractWithCustomerExcludingAssessedTax. If Gold has to reason about those raw concepts directly, the semantic complexity leaks into the analytics layer.

  • Silver absorbs source complexity: Raw SEC concepts are mapped into normalized project-level line items such as revenue, net_income, total_assets, and debt components.
  • Gold stays analytical: Once Silver presents normalized line items, Gold can focus purely on aggregation and derived ratios like gross margin, YoY growth, and debt-to-equity.
  • Maintainability: Adding new concept synonyms or fallback logic belongs in Silver, not in every downstream consumer of Gold.

Takeaway

Silver absorbs source-system mess so Gold can stay deterministic and simple.

8. Separation of Concerns Across the Medallion Layers

Decision: Each layer is responsible for exactly one class of problem.

The Engineering Rationale:

  • Bronze: "Did I successfully ingest the raw source exactly as delivered?"
  • Silver: "Did I correctly flatten, normalize, and structure the raw data into business-ready tables?"
  • Gold: "Did I compute the final metrics and ratios correctly?"

If network access, JSON flattening, and financial math all happen inside one monolithic Gold script, failures become difficult to diagnose. Splitting these responsibilities across Bronze, Silver, and Gold makes the pipeline debuggable, testable, and interview-defensible.

Takeaway

Good Medallion design is not ceremony; it is production debugging discipline.

9. Deterministic Parser First, ai_parse_document Later

Decision: Keep a deterministic custom parser for Business, Risk Factors, and MD&A in v1, and treat ai_parse_document as a future swap only after measured validation.

The Engineering Rationale:

  • Current reliability: The Spark + Python extraction path already emits the required Silver schema and integrates with existing failure logging.
  • Auditability: Regex boundaries, scoring logic, and minimum word thresholds are explicit and reproducible.
  • Controlled upgrade path: If ai_parse_document is introduced, it must be compared side-by-side on precision, boundary quality, latency, and failure rate before promotion.

Takeaway

Use the parser that is measurable and stable now; upgrade only when evidence shows a clear win.

10. Explicit Schemas Over Inference for Bronze and Silver Writes

Decision: Use strict StructType definitions for API rows, error rows, and flattened facts instead of relying on Spark schema inference.

The Engineering Rationale:

  • Type stability: SEC payload variability can cause inference drift (null-heavy batches, mixed numeric formats, optional fields).
  • Pipeline safety: Explicit schemas prevent silent type flips that can break downstream merges and aggregations.
  • Debugging speed: Data contract violations become obvious at write time instead of leaking into Gold as hard-to-trace errors.

Takeaway

In production data pipelines, schema inference is convenient for exploration but risky for contracts.

11. Canonical Accession Selection Before Metric Aggregation

Decision: Gold metrics are computed only after selecting one canonical accession per ticker-year using quality-based ranking.

The Engineering Rationale:

  • Avoid cross-accession mixing: SEC facts for the same year may appear in multiple filings or amendments.
  • Deterministic quality rules: Rank accessions by required metric hits, distinct coverage, and filing recency.
  • Metric integrity: Choose one best fact per normalized metric using unit checks, fit scores, concept priority, and recency ordering.

Takeaway

Canonical selection is the guardrail that keeps Gold financially coherent and reproducible.

12. Section Extraction: Switching from Regex to sec-parser

Decision: Replace the regex-over-flattened-text section extractor in 03_silver_decoder.py with sec-parser, a library that operates on the HTML DOM tree using iXBRL-aware element classification.

What Triggered the Decision

A head-to-head benchmark was run on the AAPL 10-K FY2024 filing (accession 0000320193-24-000123, 9.3 MB raw submission). The same filing was processed by both the current regex pipeline (output pulled from main.finsage_silver.filing_sections) and by sec-parser (run locally against the Bronze volume copy). Results were compared section by section.

The ~5% word-count reduction is not lost content. It is noise that regex includes and sec-parser correctly strips. Three specific pollution categories account for every character removed.

Pollution Type 1 — Item Heading Included in the Section Body

Regex treats the section heading as the first line of body text:

regex Business head:
  "Item 1.&#160;&#160;&#160;&#160;Business \n\n Company Background \n\n The Company designs..."

sec-parser Business head:
  "Company Background The Company designs..."

sec-parser classifies Item 1. Business as a TopSectionTitle element and starts the body at the first paragraph. That removes ~10–15 words of heading noise per section — noise that would otherwise appear verbatim at the start of every chunk fed to the embedding model.

Pollution Type 2 — Unresolved HTML Entities Embedded Mid-Text

The regex output contains literal entity strings — &#160;, &#8217;, &#8212;, &#8211; — in the text that reaches the embedding model. Four leaks were visible in the 440-character head+tail sample alone; extrapolated across 70k characters of Risk Factors, there are likely hundreds per section.

This has three concrete downstream effects:

  • "The Company's business" and "The Company&#8217;s business" embed to different vectors — the same concept produces inconsistent retrieval results.
  • Each &#160; occurrence consumes 6 token-budget characters of pure noise inside a chunk.
  • Chunkers that split on whitespace can misbehave when &#160; sits between tokens, producing unexpected split points.

sec-parser's underlying lxml/BeautifulSoup layer decodes all entities before element classification. The output contains zero entity residue.

Pollution Type 3 — Page Header and Footer Leakage

Regex has no concept of running headers or footers. When the Spark HTML stripper removes tags, the footer text that appears at the bottom of every PDF page comes through as body content:

regex MD&A tail:
  "...management's expectations could have a material impact on the Company's
   financial condition and operating results.

   Apple Inc. | 2024 Form 10-K | 2"

sec-parser MD&A tail:
  "...management's expectations could have a material impact on the Company's
   financial condition and operating results."

sec-parser classifies repeated page headers and page numbers as PageHeaderElement / PageNumberElement; they never enter the body. This matters most for RAG: the chunker may place a repeated footer like Apple Inc. | 2024 Form 10-K | 2 in the middle of a chunk, poisoning its embedding with boilerplate that will surface in unrelated similarity searches.

Content Integrity Check

The benchmark confirmed that all substantive narrative content is preserved. Section tails matched character-for-character (modulo entity decoding) across Business, Risk Factors, and MD&A:

Both end with:
  "...referenced in this Form 10-K is not incorporated by reference into this filing.
   Further, the Company's references to website URLs are intended to be inactive
   textual references only."
  (sec-parser retains the trailing period; regex cut 1 char before it — cosmetic only)

Coverage Gain Beyond AAPL

The benchmark was also run against AMZN and V, both of which produced empty Silver sections under the current regex extractor due to their post-2019 iXBRL structure. sec-parser extracted all three sections correctly for both filers.

Ticker Regex output sec-parser output Net
AAPL Works, but polluted Works, clean Quality upgrade

Why Not Keep Regex as a Fallback

Maintaining two parallel extraction paths adds branching logic, doubles the test surface, and makes Silver output non-uniform (some rows from DOM parse, some from regex). Given that sec-parser handles the AAPL case (regex's strongest example) at equal or better quality, and unblocks AMZN and V entirely, there is no coverage or quality argument for keeping the regex path in production.

Relationship to Prior Decision #9

Decision #9 ("Deterministic Parser First, ai_parse_document Later") established the principle of upgrading only when evidence shows a clear win. This benchmark is that evidence. The switch to sec-parser satisfies the three upgrade criteria set out in #9: measurable precision improvement, equal or better boundary quality, and a lower failure rate (zero iXBRL misses vs. multiple empty-section failures under regex).

Takeaway

Regex over flattened HTML extracts the right words but the wrong boundaries and includes structural noise that degrades embedding quality. sec-parser's DOM-aware element classification removes all three pollution categories with zero narrative loss, and unblocks filers whose iXBRL structure the regex approach cannot parse at all.

11. Per-Section Partial-Fallback Merge with Persisted extractor_used Provenance

Decision: After the sec-parser swap (Decision #10), 03_silver_decoder.py keeps the regex extractor as a per-section secondary tier — not a whole-filing one. sec-parser always runs first; the regex fallback runs only for required sections sec-parser missed; its output is merged into the sec-parser result by section name (sec-parser wins on ties). Each section row in silver.filing_sections carries a new extractor_used column that records which tier produced it.

This amends — rather than reverses — Decision #10. Decision #10 established sec-parser as the production extractor on clean filings, and explicitly argued against maintaining two parallel extraction paths "at the filing level". This decision does not reintroduce two paths at the filing level — it reintroduces the regex extractor at the section level, only on the sections sec-parser misses, with per-row provenance so Silver output remains auditable.

The Evidence Forcing This Amendment

The first production run of sec-parser looked clean at the surface: 30/30 tickers, 1,549 rows in filing_sections, up from 1,041. A follow-up audit of main.finsage_bronze.ingestion_errors revealed 67 extraction errors across 11 distinct filer-specific patterns. Two failure modes explained the bulk:

Failure mode Root cause in sec-parser behaviour Filers affected
Classifier misses Item 1A & Item 7 sec-parser promotes Items 1, 2, 3, 4, 5, 6 to TopSectionTitle for bank-style filers but neither TopSectionTitle nor TitleElement for Item 1A (Risk Factors) or Item 7 (MD&A). The heading text is present in the DOM; its class is just not one that _find_heading looks at. JPM, BAC, MA, PFE 10-K (7 each = 28); partial V 10-K / 10-Q.
TOC stub beats descriptive heading MSFT 10-Q places a bare TopSectionTitle("Item 2") (6 chars) ahead of the real TitleElement("ITEM 2. MANAGEMENT’S DISCUSSION AND ANALYSIS OF FINANCIAL CONDITION AND RESULTS OF OPERATIONS"). First-match-wins picks the stub; body extraction stops at the next heading (the real one), producing a near-empty body that fails the 150-word minimum. MSFT 10-Q ×19 (all quarters).

Under Decision #10's all-or-nothing fallback ("use regex only if sec-parser returned zero sections"), sec-parser's partial wins for these filers were accepted as success — Business was extracted, so the pipeline moved on — and MD&A / Risk Factors were silently lost. Worse, because silver.filing_sections is a MERGE INTO table keyed on section_id = sha256(filing_id || section_name), the regex-era rows from the previous run for those same (filing_id, section_name) pairs survived. 42 of the 1,549 silver rows (2.7%) were regex-era leftovers. Their MD&A word counts averaged 28,235 versus 9,120 for new rows — the classic signature of the old regex extractor's end-anchor failure bleeding from Item 7 through to end-of-document.

The Three-Part Engineering Rationale

(a) Why per-section merge instead of per-filing fallback

The JPM / BAC / MA / PFE failure is not "sec-parser cannot parse this filing" — it is "sec-parser parsed most of this filing correctly but missed one or two section headings". A per-filing fallback forces a choice between sec-parser's clean Business section and the regex extractor's full coverage. Per-section merge keeps both: sec-parser's entity-decoded page-footer-stripped output is preserved for every section it does classify, and the regex fallback is only enlisted for the specific sections it misses — where its page-footer and TOC-collision weaknesses do the least damage. Downstream quality impact: sec-parser's provenance stays dominant for 30+ of the 31 ticker-sections the AAPL benchmark validated, and the three pollution categories documented in Decision #10 (item-heading inclusion, HTML-entity residue, page-header/footer leakage) are cleanly eliminated on every row sec-parser sourced.

(b) Why extractor_used must be a persisted row-level column

In the first sec-parser rollout, extractor_used was tracked at the filing level inside the UDF result struct and printed as an observability counter, but it was dropped before the Silver MERGE — the column never reached the Delta table. With per-section merge, a single filing now produces a mix (sec-parser for Business, regex-fallback for MD&A), and the filing-level label would lose that distinction entirely. More importantly, persisting extractor_used per row solves the silent-staleness auditing problem: any row with extractor_used IS NULL is definitionally a row that the current extractor code did not touch — a stale survivor from a prior run. The next chunker pass can filter WHERE extractor_used IS NOT NULL to exclude pre-column rows without needing any per-run timestamp bookkeeping. The column lands on the existing table via spark.databricks.delta.schema.autoMerge.enabled = true, no rebuild required.

(c) Why the heading-preference rule is a 3-line change, not a new heuristic layer

The MSFT failure is a single, well-diagnosed bug: _find_heading returns the first regex-matching heading without regard to the heading's text length. A minimal fix — filter heading candidates whose text is ≤ 12 characters, fall back to the first if all matches are short — resolves all 19 MSFT 10-Q failures without altering behaviour for any filer whose first matching heading is already substantive. Rejected alternatives: (i) preferring the last match (risked catching back-of-document references like "Refer to Item 7 above"), (ii) ranking candidates by word count of the extracted body (worked but required running _body_between for every candidate — 4× wasted work for near-zero marginal accuracy on non-MSFT filers), and (iii) keyword-matching within heading text ("management's discussion" ⇒ MD&A) which would have required per-section keyword tables and grown the maintenance surface with no additional coverage win over the length filter on the filings that actually fail.

Relationship to Prior Decisions

Decision #3 (Never Swallow Failures): The original UDF returned an error string when a section was missing, but the downstream merge still inserted the partial-success rows into Silver and the error simply logged — making it easy to miss that MD&A was silently absent for 28 filings. The per-section merge with persisted provenance makes the failure mode visible in two places (the error log and the row itself), satisfying #3's spirit more strictly.

Decision #10 (sec-parser as Production Extractor): The coverage-and-quality arguments for sec-parser over regex on clean filings (AAPL and below) still hold — every sec-parser-sourced row on Silver is strictly higher quality than a regex-sourced one for the same section. This decision extends #10 with the observation that sec-parser's classifier is not yet reliable on all filer HTML structures, and the production shape must tolerate that by running a per-section tier-2 extractor rather than accepting sec-parser's partial coverage as final.

Deferred

Takeaway

A production extractor must degrade gracefully at the section level, not the filing level, and every section row must record which extractor produced it. Without per-section fallback, sec-parser's partial wins silently dropped required sections for bank-style filers; without per-row provenance, regex-era rows survived a merge-based silver rebuild and inflated MD&A embeddings by 3×. Fixing both — plus a three-line heading-preference rule for MSFT's TOC-stub pattern — dropped the filer-specific error surface from 67 to ~15 and replaced "30/30 tickers" with a number that is actually audit-trail-defensible.

13. Multi-Dimensional Eval over a Single LLM-as-Judge

Decision: The agent evaluation harness in 07_evaluation.py runs eight independent scorers per question against the 100-question stratified ground-truth set, not a single "is the answer right?" LLM judge. Each scorer measures one specific dimension (correctness, citation grammar, refusal-reason validity, tool routing, numerical tolerance, derived-metric tolerance, retrieval grounding, ticker+year citation), can SKIP rows that aren't applicable, and persists its outcome row-level to main.finsage_gold.eval_question_outcomes so failures are diagnosable by dimension rather than as a single fuzzy "looks bad" verdict.

Why a Single LLM Judge Is the Wrong Abstraction

The default temptation when evaluating an LLM agent is to pose the question to another LLM ("here's the agent's answer, here's the expected answer — same?") and aggregate to a percentage. This collapses every failure mode into one channel and produces three predictable pathologies:

  • Phrasing-vs-correctness confusion. The judge can't reliably distinguish "the answer is mathematically wrong" from "the answer is right but uses different precision ($391.04B vs $391,035,000,000)". Both surface as a binary fail; you can't tell which without re-reading every trace.
  • No notion of applicability. A refusal-test question and a numerical-lookup question fail for completely different reasons. Asking the same judge to score both with one rubric forces it into mushy generalities ("the answer is reasonable") that don't catch real regressions.
  • Over-strict on precision, under-strict on grounding. Real failure modes — agent cited a different ticker than the one retrieved, agent volunteered a substitute number when the requested metric was N/A, agent used the annual tool for a quarterly question — all blur under a single "correctness" verdict.

The Eight Scorers, Their Domain, and Their Math

Scorer Type What passes When it SKIPs
correctness LLM judge (built-in) llama-3.3-70b says the response is semantically equivalent to expected_response. Never — applies to every row.
cites_ticker_and_year LLM judge (Guidelines) The judge confirms the response cites both a ticker symbol and a fiscal year when discussing any financial figure. Never — applies to every row.
numerical_tolerance Custom @scorer B/M/K-aware extraction of the first numeric figure from response vs expected_response; relative error ≤ 1%. category != numerical_lookup.
derived_metric_match Custom @scorer Same number extraction, but applied to derived ratios (margins, growth rates, D/E). Tolerance is max(rel_err ≤ 1%, abs_err ≤ 0.005) — the absolute floor matters for small ratios where 1% relative is noisy. category not in {numerical_lookup, yoy_comparison, multi_company}.
citation_format Custom @scorer Response contains both a [VERBATIM]/[SUMMARY] tag and a [Source: TICKER | FY####...] line. source_section in {metrics, 10-K Cover Page, ""} (deterministic answers don't quote retrieved text).
refusal_correctness Custom @scorer Response contains a refusal phrase from a broad whitelist (modern negation+auxiliary forms included: "isn't", "hasn't occurred", "couldn't find", etc.) AND mentions at least one expected reason token from _REFUSAL_CONTEXT_TOKENS[qid]. category != refusal_test.
tool_routing_correctness Custom @scorer Response's source-line shape matches the expected tool for that question: FY#### | metrics for annual, FY#### Q# for quarterly, 10-K Cover Page for filing metadata, 10-K | Section or 10-Q | Section for retrieval. Refusal-test rows (no tool expected); rows where category doesn't constrain a single tool.
retrieval_grounded_when_used Custom @scorer Trace contains a RETRIEVER span AND the response cites a ticker that overlaps with at least one retrieved chunk's ticker. No RETRIEVER span in the trace (agent answered without invoking search_filings) — this is the SKIP-not-ERROR replacement for MLflow's built-in RetrievalGroundedness, which hard-errors on traces with no retrieval.

Why SKIP, Not FAIL, on Inapplicability

The default MLflow GenAI scorer pattern returns True/False. Returning None is the documented signal that the row is not applicable to this scorer. We use it aggressively. The reason is statistical — if numerical_tolerance ran on all 100 questions and forced a False on the 80 questions that aren't numerical lookups, the headline pass-rate would be a meaningless 20%. With SKIP, the pass-rate denominator is the 20 applicable rows, and a 19/20 PASS reads honestly as 95% on the dimension that scorer measures. Equally important: a SKIP doesn't pollute the regression diff. When we compare v20 → v21, we want to see "20/20 → 18/20 on numerical_tolerance" — not 18/20 dragged into a 100-row average where 80 of the rows are noise.

Why Sum-of-Fails Across Scorers ≠ Number of Broken Questions

A single bad answer typically trips multiple scorers at once. Question A014 (BAC FY2022 total_liabilities) failed in the v21 run on correctness, cites_ticker_and_year, AND numerical_tolerance — three fails from one root cause (the agent's get_company_metrics tool didn't display total_liabilities in its formatted output). When that single fix landed, all three fail counts drop together. This is a feature, not a bug: it confirms the scorers measure independent dimensions of the same underlying answer quality, and lets us attribute fail-count deltas to specific code changes. A single LLM judge would have just nudged its score down by an arbitrary amount with no diagnostic signal.

Persistence Schema and Why It's Delta, Not MLflow-Only

Every scorer outcome lands in main.finsage_gold.eval_question_outcomes with columns (run_id, question_id, scorer_name, category, ticker, fiscal_year, fiscal_quarter, difficulty, outcome, value_numeric, rationale, error_message, agent_response, expected_response, recorded_at), partitioned by run_id. The aggregate per-run row lands in eval_run_summaries with metrics_json, agent_version, dataset_hash, git_commit, mlflow_url. Both tables MERGE on run_id — idempotent on re-runs of the persistence cell. The reason for Delta over relying solely on MLflow's experiment store: MLflow assessments are scoped to one run at a time and have no SQL surface. We want regression-diff queries like "show me every run since v15 where numerical_tolerance/pass_rate < 0.85, ordered by recorded_at" — that's one Delta query, not a chain of MLflow REST calls.

The Trade-off We Accepted

  • More scorers means more wall-clock per run. Eight scorers × 100 questions × LLM-judge round-trips for the LLM-judge subset gives ~5 minutes per run. Acceptable: we run eval on demand after a redeploy, not in a hot loop.
  • More scorers means more code to maintain. Mitigated by the unit-test gate — tests/unit/test_scorers.py + test_eval_preflight.py total 67 cases that catch scorer-wiring regressions in <1s locally.
  • The LLM-judge scorers (correctness, cites_ticker_and_year) can still be wrong both ways. They're useful as the wide semantic check but should never be the sole gate. Custom math scorers (numerical_tolerance, derived_metric_match) are the deterministic second opinion that prevents an LLM-judge whim from masking a real regression.

Relationship to Prior Decisions

Decision #3 (Never Swallow Failures): The eight-scorer architecture is the runtime expression of #3 — every failure mode has a named channel, no silent regressions. Decision #6 (XBRL as the Numerical System of Record): numerical_tolerance and derived_metric_match compare against XBRL-sourced gold values, not LLM-derived ones — the eval is itself anchored on XBRL provenance. Decision #11 (Per-Section Fallback Merge): the extractor_used column we added there is exactly the kind of row-level provenance that the persistence schema mirrors here — a fail row in eval_question_outcomes always carries enough metadata to drill back to the underlying question and trace.

Takeaway

A single LLM-as-judge is the easy way to evaluate an agent and the wrong way to improve one. It collapses orthogonal failure modes into one fuzzy verdict and gives no diagnostic signal when the score moves. Eight independent scorers, each with a named domain and an honest SKIP for inapplicable rows, turn the eval from "is the agent good?" into "on which dimension did this build regress, by how much, on which questions, and which root cause does that imply?" — and that turns the eval into something an engineer can actually act on.
