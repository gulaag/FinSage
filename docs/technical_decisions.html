Technical Decisions

This document captures the strategic "why" behind the FinSage architecture, specifically outlining why certain Medallion and Agent design choices were made over alternative approaches.

1. Scope: 30 Companies (Depth Over Breadth)

Decision: The platform is explicitly constrained to 30 Tier-1 companies (yielding ~730 filings total), rather than attempting to scrape the entire Russell 3000 or S&P 500.

The Engineering Rationale:

  • Deterministic Validation: The core of this project is proving numerical precision and extraction accuracy (the "North Star Metric"). With 30 companies, I can manually open an SEC PDF, look at the Balance Sheet, and verify that my pipeline extracted the exact same number. You cannot manually audit an automated pipeline scraping 3,000 companies, which leads to silent failures and "hallucinated" downstream analytics.
  • Evaluation Rigor: I am building a 100-question evaluation suite to test the RAG Agent. Ensuring those 100 questions have perfectly accurate ground-truth answers requires a tightly controlled dataset.
  • Infrastructure Agility: Processing 30 companies allows me to iterate on pipeline logic, test new chunking strategies, and run the entire Medallion architecture on a cost-effective Databricks cluster in minutes.
  • Scalability is a Config Change: Because the foundation is built on Auto Loader, Unity Catalog, and Delta Lake, the architecture is already inherently distributed. Scaling to 300 companies later requires zero code rewrites—it only requires increasing the cluster size and updating the target ticker list.

2. Decoupled Parsing: "Native JVM First, Python Last"

Decision: Perform all heavy document truncation and initial cleaning using Spark's native SQL functions, and restrict Python UDFs to lightweight section slicing.

The Engineering Rationale:

Initially, I attempted to pass the raw, 50MB binary content from the Bronze layer directly into a Python UDF to be parsed by BeautifulSoup. This caused massive Py4J serialization overhead and crashed the worker nodes.

By decoupling the work, I leverage the exact strengths of the Databricks engine:

  • Spark SQL (JVM): Handles the brute-force work. substring_index and regexp_replace slice off the massive XBRL/Image data dumps and strip HTML tags instantly across distributed nodes.
  • Python UDF: Handles the surgical work. It receives a tiny, 1MB string and uses Python's robust re library to identify "Item 1A" and "Item 7" section boundaries.

This architectural pivot reduced processing time per document from minutes (crashing) to milliseconds (stable).

3. Error Auditing: The "Never Swallow Failures" Constraint

Decision: The text-extraction UDF must never crash the pipeline or silently drop records when it encounters a malformed SEC filing. It must route bad data to a dedicated audit table.

The Engineering Rationale:

In real-world data engineering, source data is chaotic. Older SEC filings, amended 8-Ks, or filings from specific sectors often do not contain standard "Item 1A" or "Item 7" string headers.

Instead of writing fragile try/except blocks that just skip the file, I designed the UDF to return a complex StructType containing both the sections array and an error string flag.

  • If a document parses perfectly, the error is null, and the sections are exploded and merged idempotently into the Silver table.
  • If a document fails, the pipeline catches the error string, attaches the file_path and a failed_at timestamp, and writes it directly to main.finsage_bronze.ingestion_errors.

This pattern guarantees high pipeline uptime while giving me a queryable "to-do list" of edge cases to investigate and fix in future sprint cycles, perfectly adhering to the Blueprint 4.1 compliance standards.
